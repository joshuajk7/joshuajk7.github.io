Project Overview

BigShell is a lightweight command-line interpreter that implements a meaningful subset of POSIX shell functionality. It demonstrates core systems programming concepts: process creation and control, signal handling, input/output redirection, job control, and environment/variable handling.

Why it matters

This project showcases practical experience with low-level system calls and concurrency primitives used in UNIX-like operating systems. It’s a strong example of systems-level programming and debugging in a production-like workflow.

Core features

  • Interactive REPL with command parsing and comments/escaping support
  • Builtin commands: cd, exit
  • I/O redirection: <, >, >>
  • Background jobs with & and job tracking
  • Variable and parameter expansion: $$, $!, $?, ${VAR}
  • Signal handling (SIGINT, SIGTSTP) and robust child management

Project structure

The codebase is separated into logical modules for maintainability and clarity:

  • bigshell.c — main REPL, startup, and loop
  • parser.c — tokenization and parsing of commands
  • runner.c — execution, I/O redirection, and builtins
  • jobs.c — job control and tracking
  • signals.c — signal setup, handlers, and restore logic
  • vars.c — environment and shell variable management

Build & run

Typical build / run commands used during development:

make
# or to compile directly
gcc -Wall -Wextra -std=gnu11 -o bigshell *.c

Testing & tools

Development used Git for version control, a Makefile for builds, and GDB for debugging. Automated grading scripts (GradeScope) were used during assignment submission.

Implementation highlights

Selected design decisions that highlight engineering judgment:

  • Modular code layout to separate parsing, execution, and signal handling — easier to test and debug.
  • Explicit environment variable handling so child processes receive exported variables.
  • Careful use of waitpid with flags WNOHANG and WUNTRACED to track background/foreground and stopped children.
  • Signal masks and sigaction usage to prevent the shell from exiting while delivering signals to child processes.

Challenges & lessons

Key challenges included:

  • Managing job control and correctly restoring process group and signal state.
  • Debugging tricky race conditions with child processes and signal delivery.
  • Time management and iterative testing — breaking the project into smaller deliverables helped.

Lessons learned: plan early, test incrementally, and rely on system debuggers (GDB) when dealing with memory or control-flow issues.

Conclusion & next steps

BigShell meets the assignment's goals and serves as a foundation for additional features. Potential improvements for future versions include:

  • Adding pipeline support (|) and shell scripting constructs.
  • Improving job control builtins (e.g., jobs, fg, bg).
  • Enhanced error reporting and unit tests for parser components.
